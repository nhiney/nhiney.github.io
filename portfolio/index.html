Behind the code

Identity

Engineering, with intent.

I'm Nhi — a backend-leaning engineer who cares more about how a system ages than how it launches. This page is not a CV or a project list; it's the bit of me that doesn't fit into either.

01 · Principles

What I keep returning to

Habits I've built from getting things wrong, often. They show up in my pull requests more than in my résumé.

01

Read the code before you write it

Most bugs aren't bugs — they're misread context. I spend more time understanding than typing.

02

Logs are louder than guesses

If a system can't explain itself at 2 a.m., it isn't finished. I optimise for observability before cleverness.

03

Boring shipped beats brilliant blocked

Predictable code that ships unblocks everyone else. The fancy refactor can wait for a sober Tuesday.

04

Security is hygiene, not a feature

Threat modelling lives at the requirements stage, not in a final-week 'security review'.

02 · How I work

The shape of a good day

Process is just the part of taste that's repeatable.

Planning

I ask why before how. A small written one-pager beats an hour of meetings.

Communication

Async-first, doc-heavy. If a decision is worth making, it's worth writing down.

Code review

Slow, careful, and kind. Reviews teach the team — they aren't a gate.

Shipping

Small commits, clear messages, tests where they matter. Done > perfect.

03 · Currently exploring

What's on my radar this season

A small, honest list — not everything I've ever clicked.

  • Workflow

04 · Beyond the code

What fuels the engineer

Tools I happen to use; values I actually live by.

Reading

Engineering essays, slow non-fiction, anything that respects the reader's time.

Quiet mornings

First two hours, no Slack. The work I'm proud of always starts there.

Mentoring

Pairing with juniors. Half the time I'm the one who learns something.

Coffee, walking, journaling

Boring fuel for non-boring work. I'd rather be sustainable than fast.

05 · Heading toward

The kind of work I want next

Teams that respect users and reviewers in equal measure. Backends that are honest about their failure modes. A practice that can survive a decade — mine, and the codebase's. I'd rather be the careful engineer than the fast one, and I think those two things only feel opposed if the team is in a hurry for the wrong reasons.

Building something quiet but excellent?

I'd love to hear about it. Whether it's a backend that needs care, a team that values craft, or just a conversation about engineering done well — my inbox is open.
